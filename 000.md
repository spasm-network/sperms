# SRM-000

## How to create SRM

`draft` `optional` `meta` `guardian:spasm-core`

This SRM defines the standard format and process for writing and submitting Spasm Protocol Ecosystem Reference Module.

---

### What is a SRM?

**Spasm Protocol Ecosystem Reference Module** is a design document describing a new feature, module, protocol change, or convention for the Spasm ecosystem. SRMs are the primary mechanism for proposing and documenting changes.

---

### SRM Status Tags

Status tags are included in the header:

| Tag | Meaning |
|-----|---------|
| `draft` | Work in progress, not yet finalized |
| `proposed` | Ready for community review |
| `active` | Accepted and in use |
| `deprecated` | No longer recommended |
| `mandatory` | Implementations must support this to stay compatible |
| `optional` | Implementations may support this |
| `guardian:name`| SRM's guardian, allowed multiple (optional) |

---

### SRM Structure

1. **Title** — SRM count (e.g. `SRM-01`)
2. **Subtitle** — One sentence describing the module's purpose
3. **Status tags** — As defined above
4. **Abstract** — A brief summary of what the SRM proposes
5. **Motivation** — Why this change or feature is needed
6. **Specification** — The detailed technical description with examples
    - Use sub-sections (###) for major components.
    - Include code blocks, diagrams, or tables where necessary.
    - If applicable, reference the relevant SpasmEventV2 interfaces defined in the types.
7. **Backwards Compatibility** — Whether this SRM breaks existing behavior, and how to handle it
8. **References** — Links to related SRMs or external documents (optional)

---

### Numbering

- `SIP-00` (this document) defines the meta-format
- Numbers are assigned at submission time and must not be reused

---

### Submission Process

1. Write your SRM following this format
2. Open a pull request to the SRMs repository with the file named `SRM-XX.md`
3. The community reviews and discusses the proposal
4. Maintainers update the status tag upon acceptance

When SRM is merged, it becomes the official standard. However, unmerged SRMs can still be used and implemented by network participants as unofficial standards.

---

### Submission Checklist

Before submitting your PR:
- [ ] Is the header `# SRM-XXX` present and correctly numbered?
- [ ] Are all required status tags included in the subtitle line?
- [ ] Is the Abstract short and clear?
- [ ] Is the Specification section technically accurate according to the latest Spasm types?
- [ ] Is your module backwards-compatible?
- [ ] Are all links in the References section valid?
