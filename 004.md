# SRM-004

## SpasmEventV2 Universal Wrapper

`draft` `mandatory` `guardian:spasm-core`

### Abstract
This document defines the `SpasmEventV2` interface as the universal, deterministic wrapper for various events. It specifies how disparate messaging formats (Spasm, Nostr, DMP, RSS) are encapsulated into a single, standardized JSON object. This ensures that developers do not need to maintain separate UIs or database schemas for different protocols, allowing them to work with a single `SpasmEventV2` type regardless of the underlying format.

### Motivation
Currently, the Spasm ecosystem supports multiple underlying protocols, which creates architectural complexity:
1.  **Fragmented Frontends:** Developers often build three separate UIs, one for each protocol version.
2.  **Fragmented Databases:** Three separate database tables or schemas are required to store events from different sources.
3.  **Serialization Chaos:** Events from different protocols have different IDs.

The `SpasmEventV2` wrapper abstracts these differences. It enforces a strict schema where:
-   The `type` field explicitly identifies the wrapper variant (e.g., `SpasmEventV2`, `SpasmEventEnvelopeV2`, `SpasmEventDatabaseV2`).
-   Protocol-specific data resides in the `siblings` array.
-   Signatures can be multiple, allowing one event to be signed by different keys/protocols while maintaining the same Spasm ID.

### Specification

The `SpasmEventV2` system consists of a hierarchy of interfaces designed to separate the "Event Content" from the "Envelope/Context" and "Storage/Tree" metadata.

#### Types
All events in the ecosystem must be converted into one of the following types before being processed or stored:

```typescript
export type SpasmEventAnyV2 = 
  | SpasmEventV2
  | SpasmEventBodyV2
  | SpasmEventEnvelopeV2
  | SpasmEventEnvelopeWithTreeV2
  | SpasmEventDatabaseV2
  | SpasmEventBodySignedClosedV2
```

#### The Core Wrapper: `SpasmEventV2`

This is the primary interface exposed to applications. It omits protocol-specific details that vary depending on the siblings but typicaly includes authors, title, content, IDs.

TypeScript interface from [spasm.js](https://www.npmjs.com/package/spasm.js) npm library.

```typescript
export interface SpasmEventV2 extends
  Omit<SpasmEventBodyV2,
    'type' | 'protocol' | 'authors' | 'parent' | 'hosts' | 'links' | 'references' | 'sequence' | 'previousEvent'>,
  // Omit<SpasmEventEnvelopeV2, 'type'>,
  Omit<SpasmEventEnvelopeWithTreeV2,
    'type' | 'protocol' | 'authors' | 'parent' | 'root' | 'hosts' | 'links' | 'references' | 'sequence' | 'previousEvent' | 'children'> {
  type: "SpasmEventV2"
  authors?: SpasmEventAuthorV2[]
  root?: SpasmEventRootV2
  parent?: SpasmEventParentV2
  hosts?: SpasmEventHostV2[]
  links?: SpasmEventLinkV2[]
  references?: SpasmEventReferenceV2[]
  // Some events don't have signatures (e.g. RSS posts, URLs)
  signatures?: SpasmEventSignatureV2[]
  children?: SpasmEventChildV2[]
}
```
